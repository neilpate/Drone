# ADR 0003 — IMU: ICM-42688-P on SPI

- **Status:** Accepted
- **Date:** 2026-05-21
- **Related:** [ADR 0001](0001-platform-airframe-stack.md), [ADR 0002](0002-mcu-and-language.md)

## Context

[ADR 0002](0002-mcu-and-language.md) committed us to the micro:bit v2 for Phases 1–3. The board's onboard sensor (LSM303AGR) is accelerometer + magnetometer only — **no gyroscope**. A quadcopter cannot stabilise without gyro data, so an external 6-DoF IMU is mandatory before any meaningful sensor / fusion / control work begins.

Candidates considered (full comparison in chat history): MPU-6050 (old, cheap, EOL clone-land), MPU-9250 / ICM-20948 (older, 9-DoF), **ICM-42688-P** (modern, what Betaflight uses), BMI270 (modern Bosch equivalent), LSM6DSOX (modern ST).

## Decision

- **Part:** TDK InvenSense **ICM-42688-P** (6-DoF: 3-axis gyro + 3-axis accel).
- **Bus:** **SPI** (mode 0, up to 24 MHz per datasheet — we'll start much slower).
- **Form:** off-the-shelf breakout board with 0.1″ headers. Specific vendor TBD at purchase time (Pimoroni in UK is the likely source).
- **Magnetometer:** **not used for now.** The micro:bit's onboard LSM303AGR mag remains available "for free" later if heading becomes useful (it almost never does for stabilisation; it matters for autonomous nav).
- **Interrupt line:** ICM `INT1` → micro:bit GPIO, used for data-ready. No polling.

## Why this part, this bus

- **ICM-42688-P is the current standard** on modern hobby flight controllers (Betaflight default for a couple of generations). Whatever we learn carries over directly to a real FC board at Phase 4+.
- **Excellent noise performance** (~0.0028 °/s/√Hz gyro noise density) vs. MPU-6050 (~0.05). Means a fusion filter that *should* work won't be sabotaged by the sensor.
- **SPI over I²C** because:
  - Real flight controllers all use SPI for the gyro — control loops eventually run at 4–8 kHz, well past I²C's comfort zone.
  - SPI is full-duplex, deterministic timing, no bus arbitration.
  - Learning SPI properly (CPOL/CPHA, CS handling, DMA later) is on the learning checklist anyway.
  - I²C on the micro:bit is already shared with the onboard LSM303AGR — keeping the new sensor on its own bus avoids weird interactions while we're learning.

## Consequences

### What this commits us to

- **An external IMU breakout board, edge-connector breakout, breadboard, jumpers.** ~£25–30 total. See chat / future shopping-list doc.
- **Writing or adopting a Rust driver for ICM-42688.** A community `icm42688` crate exists and is the starting point; we accept we may end up writing register-level code ourselves, which is fine — that's part of the learning.
- **Wiring commitment** (subject to revision when actually wired):
  - SCLK → P13, SDO/MISO → P14, SDI/MOSI → P15, CS → P16, INT1 → P8, VDD/VDDIO → 3V, GND → GND.
  - The micro:bit's conventional SPI pin assignment uses P13/P14/P15; CS and INT pins are free choice.
- **An interrupt-driven sampling architecture** from the start. INT1 wakes the MCU; sample is read over SPI; pushed into the fusion loop. `embassy` async fits this naturally.

### What this rules out (for now)

- **No magnetometer in the fusion loop** for early phases. Attitude estimate will rely on gyro (high-frequency) + accel (low-frequency tilt reference) only — a 6-DoF complementary or Mahony filter, not a 9-DoF one.
- **No I²C path to this IMU.** If SPI causes us serious trouble we revisit, but committing to one bus removes a variable.
- **No fancy on-chip features** (FIFO, ML core in other parts, on-chip DMP). We do the fusion ourselves on the MCU — that's the point.

### What stays open

- Specific breakout vendor + part number → resolved at purchase, recorded in a shopping-list / BOM doc when we have one.
- SPI clock speed in production — start conservative (e.g. 1 MHz) for bring-up, push later.
- Whether we eventually add a second IMU for redundancy / cross-check. Not on the roadmap for now.
- Magnetometer use (onboard LSM303AGR) if/when we get to heading-aware phases.
